# 系统容量测试方法套路

## 1. 文档目标

本文档用于指导系统容量测试从准备、执行、分析到生产容量结论落地。

容量测试的目标不是打出最高 QPS，而是确定：

```text
在目标业务场景下，系统满足错误率、延迟、一致性和资源水位要求时，可以长期稳定承载的最大吞吐。
```

最终产出必须能回答：

```text
1. 当前系统在不同业务场景下的有效容量是多少；
2. 哪个环节最先成为瓶颈；
3. 生产建议安全水位是多少；
4. 流量超过安全水位后如何扩容、限流或降级；
5. 上线前是否满足容量准入标准。
```

***

## 2. 容量口径

### 2.1 三个容量数字

容量测试必须区分三个数字。

|容量类型|含义|用途|
|---|---|---|
|峰值容量|短时间内系统能打到的最高吞吐|只用于了解极限，不作为生产承诺|
|有效容量|满足错误率、P95、P99、一致性要求的最大稳定吞吐|容量测试的核心结论|
|生产安全水位|建议生产长期运行的吞吐上限|用于容量规划、告警、限流和扩容|

推荐关系：

```text
生产安全水位 = 有效容量的 50% ~ 70%
```

如果系统依赖 MQ、异步任务、补偿任务、最终一致性链路，安全水位应更保守。

### 2.2 容量结论必须绑定场景

错误写法：

```text
系统容量是 100 QPS。
```

正确写法：

```text
在分散写场景下，系统有效容量约 80 RPS，生产安全水位建议 50 RPS。
在热点 Gift 场景下，系统有效容量约 30 RPS，生产安全水位建议 20 RPS。
在热点用户场景下，系统有效容量约 15 RPS，生产安全水位建议 10 RPS。
```

同一个系统在不同数据分布、热点比例、业务成功率下，容量会完全不同。

### 2.3 总容量定义

系统总容量不能简单取所有场景中最高的值。

生产容量应按真实流量结构计算：

```text
总容量 = 各业务场景容量约束下，系统整体能稳定承载的混合流量。
```

如果线上存在明显热点，应按热点场景约束设计。

例如：

```text
普通分散写容量：80 RPS
热点 Gift 容量：30 RPS
热点用户容量：15 RPS
极端双热点容量：8 RPS
```

如果大促期间可能出现双热点，则生产保护策略必须按双热点 8 RPS 附近设计，而不是按普通分散写 80 RPS 设计。

## 3. 容量测试准入条件

没有准入条件的容量测试结果不可用于生产决策。

### 3.1 环境准入

容量测试环境应尽量接近生产。

最低要求：

```text
1. 应用配置与生产一致或差异可解释；
2. JVM 参数与生产一致或差异可解释；
3. MySQL / Redis / MQ 规格明确；
4. 网络链路明确；
5. 应用实例数明确；
6. 日志级别符合压测要求；
7. 数据库连接池、线程池、队列配置明确；
8. 监控、日志、链路追踪可用。
```

禁止在以下状态做容量结论：

```text
1. SQL debug 日志大量输出；
2. 测试数据和生产量级差异巨大且未说明；
3. 压测机和被测服务抢占同一台机器资源但未记录；
4. 监控缺失，只保留 k6 输出；
5. 压测期间有人手工操作数据库或重启服务；
6. 业务失败和系统失败混在一起统计。
```

### 3.2 数据准入

容量测试前必须准备固定数据集。

必须明确：

```text
1. 用户数量；
2. 礼物数量；
3. 直播间数量；
4. 每个用户初始余额；
5. 每个礼物初始库存；
6. 热点用户列表；
7. 热点礼物列表；
8. requestId 生成规则；
9. 数据重置方式；
10. 压测后一致性校验 SQL。
```

### 3.3 观测准入

容量测试过程中至少要能观察：

```text
1. 请求吞吐：RPS / TPS；
2. 延迟：avg、P90、P95、P99、max；
3. 错误：HTTP 5xx、超时、连接失败、业务失败；
4. 应用：CPU、内存、GC、线程、队列；
5. 连接池：active、idle、pending、timeout；
6. MySQL：QPS、慢 SQL、锁等待、连接数、事务；
7. Redis：QPS、延迟、慢命令、内存、连接数；
8. MQ：生产速率、消费速率、堆积量、消费延迟；
9. 一致性：订单、余额、库存、流水、补偿状态。
```

## 4. 容量测试标准流程

### 4.1 第一步：定义业务容量目标

先写清楚目标，不直接开压。

模板：

```text
业务场景：
目标峰值流量：
目标平均流量：
目标持续时间：
请求成功定义：
可接受业务失败：
不可接受系统失败：
P95 阈值：
P99 阈值：
错误率阈值：
一致性要求：
```

示例：

```text
业务场景：直播间送礼
目标峰值流量：50 RPS
目标持续时间：30 分钟
请求成功定义：订单创建成功，余额扣减成功，库存扣减成功
可接受业务失败：库存不足、余额不足
不可接受系统失败：HTTP 5xx、超时、数据库连接池超时
P95 阈值：3000ms
P99 阈值：5000ms
系统错误率阈值：< 1%
一致性要求：不超卖、不超扣、不重复订单
```

### 4.2 第二步：建立场景矩阵

至少覆盖：

|场景|目的|
|---|---|
|分散写|验证普通容量|
|热点资源|验证库存、账户、房间等热点容量|
|混合流量|验证真实流量结构下的总容量|
|突刺流量|验证短时洪峰保护能力|
|长稳流量|验证长期运行稳定性|

Gift 项目建议矩阵：

|场景|用户分布|礼物分布|验证目标|
|---|---|---|---|
|分散写|1000 用户|100 礼物|普通交易容量|
|热点 Gift|1000 用户|1 礼物|库存热点容量|
|热点用户|5 用户|100 礼物|账户热点容量|
|双热点|5 用户|1 礼物|极限热点容量|
|混合流量|按生产比例|按生产比例|总容量评估|

### 4.3 第三步：冒烟压测

目的：

```text
确认脚本、数据、接口、成功判断、一致性校验都是正确的。
```

建议参数：

```text
并发：1 ~ 5 VU
持续时间：1 ~ 3 分钟
```

通过标准：

```text
1. 请求能正常返回；
2. k6 checks 正确区分成功、业务失败、系统失败；
3. requestId 唯一；
4. 数据库订单、余额、库存变化符合预期；
5. Redis 库存、库存流水状态符合预期。
```

### 4.4 第四步：基准容量测试

目的：

```text
得到当前版本在各核心场景下的基础容量。
```

建议参数：

```text
并发：50 VU、100 VU
持续时间：5 ~ 10 分钟
```

输出：

```text
1. 每个场景 RPS；
2. avg / P90 / P95 / P99；
3. 系统错误率；
4. 业务失败率；
5. 一致性校验；
6. 初步瓶颈。
```

### 4.5 第五步：阶梯加压找容量拐点

目的：

```text
找到系统从稳定进入排队、超时、错误的临界点。
```

推荐模型：

```text
20 RPS -> 40 RPS -> 60 RPS -> 80 RPS -> 100 RPS
每档持续 5 分钟
每档记录指标
```

也可以使用 VU 阶梯：

```text
20 VU -> 50 VU -> 100 VU -> 200 VU -> 400 VU
每档持续 5 分钟
```

更推荐生产容量评估使用固定到达率 RPS，因为它更接近入口流量。

容量拐点判断：

```text
1. 压力增加，RPS 不再上升；
2. P95 / P99 快速上升；
3. 系统错误率开始上升；
4. Hikari pending 开始出现；
5. DB 锁等待开始出现；
6. Redis 延迟开始升高；
7. MQ 或本地队列开始持续堆积；
8. CPU、IO、网络任一资源接近瓶颈。
```

容量取法：

```text
有效容量取容量拐点前一档。
```

### 4.6 第六步：有效容量复测

阶梯加压只能找到大致边界，不能直接作为生产容量。

找到拐点后，需要围绕拐点复测：

```text
如果 80 RPS 稳定，100 RPS 开始恶化，则复测：
70 RPS
80 RPS
90 RPS
```

每档持续：

```text
10 ~ 30 分钟
```

通过标准：

```text
1. 错误率满足阈值；
2. P95 / P99 满足阈值；
3. 资源水位没有持续上升；
4. 队列没有持续堆积；
5. 异步任务可以追平；
6. 一致性校验通过。
```

### 4.7 第七步：长稳容量验证

目的：

```text
验证有效容量在较长时间内是否稳定。
```

推荐压力：

```text
有效容量的 60% ~ 80%
```

推荐持续时间：

```text
30 分钟、1 小时或更长。
```

重点观察：

```text
1. P95 / P99 是否随时间变差；
2. JVM 内存是否持续增长；
3. GC 是否越来越频繁；
4. DB 连接是否回收正常；
5. Redis 内存和延迟是否稳定；
6. MQ / 本地队列是否可追平；
7. 补偿任务是否长期堆积；
8. 业务数据是否一致。
```

### 4.8 第八步：突刺和恢复测试

目的：

```text
验证流量洪峰下系统是否能保护核心链路，并在洪峰后恢复。
```

推荐模型：

```text
正常流量 5 分钟
突增到 3 倍流量 1 分钟
回落到正常流量 10 分钟
```

通过标准：

```text
1. 突刺期间核心服务不崩溃；
2. 允许限流或业务失败，但系统失败要可控；
3. 队列不无限增长；
4. 回落后 P95 / P99 恢复；
5. 异步任务和补偿任务能追平；
6. 数据一致性最终通过。
```

## 5. 容量计算方法

### 5.1 单场景有效容量

公式：

```text
单场景有效容量 = 满足 SLO 和一致性要求的最大稳定 RPS
```

示例：

```text
60 RPS：P95 2.5s，P99 4.2s，错误率 0%，一致性通过
80 RPS：P95 3.1s，P99 5.8s，错误率 0.3%
100 RPS：P95 8s，P99 15s，错误率 3%
```

如果 SLO 是：

```text
P95 < 3s
P99 < 5s
错误率 < 1%
```

则有效容量应取：

```text
60 RPS
```

### 5.2 混合流量总容量

混合流量应按业务比例测试，不用单场景容量简单相加。

示例生产流量结构：

```text
普通分散写：70%
热点 Gift：20%
热点用户：8%
双热点：2%
```

测试方式：

```text
用一个混合脚本按上述比例发流量，阶梯加压寻找总容量。
```

结论示例：

```text
在 70% 分散写、20% 热点 Gift、8% 热点用户、2% 双热点的混合流量下，
系统有效容量为 45 RPS，生产安全水位建议 30 RPS。
```

### 5.3 多实例容量

多实例容量不能默认线性增长。

测试方式：

```text
1 个实例测一次；
2 个实例测一次；
4 个实例测一次；
```

计算扩展效率：

```text
扩展效率 = 多实例有效容量 / (单实例有效容量 * 实例数)
```

示例：

```text
单实例有效容量：50 RPS
双实例有效容量：85 RPS
扩展效率 = 85 / (50 * 2) = 85%
```

判断：

```text
1. 扩展效率接近 100%：应用基本可水平扩展；
2. 扩展效率明显低于 70%：共享资源成为瓶颈；
3. 多实例容量不升反降：DB、Redis、锁、热点 Key、全局队列需要重点排查。
```

### 5.4 生产安全水位

建议计算：

```text
生产安全水位 = 有效容量 * 安全系数
```

安全系数建议：

|系统状态|安全系数|
|---|---:|
|纯同步链路，瓶颈明确，恢复简单|0.7|
|有 Redis / DB 共享资源热点|0.6|
|有 MQ / 异步 / 补偿 / 最终一致性|0.5|
|大促、活动、不可预测热点|0.3 ~ 0.5|

示例：

```text
有效容量：80 RPS
系统依赖 Redis 预扣和库存补偿
安全系数：0.5
生产安全水位：40 RPS
```

## 6. 生产落地判定标准

### 6.1 Go / No-Go 标准

上线前必须满足：

```text
1. 目标混合流量达到业务目标；
2. P95 / P99 满足阈值；
3. 系统错误率满足阈值；
4. 长稳测试通过；
5. 突刺后系统可恢复；
6. 数据一致性校验通过；
7. 队列、补偿、同步任务不长期堆积；
8. 已配置容量告警；
9. 已配置限流或降级策略；
10. 已明确扩容方式和回滚方式。
```

No-Go 条件：

```text
1. 容量结论只来自一次短压；
2. 没有 P95 / P99；
3. 没有错误分类；
4. 没有一致性校验；
5. 资源水位持续上升但未解释；
6. 异步任务持续堆积；
7. 数据存在超卖、超扣、重复订单；
8. 不知道瓶颈在哪；
9. 没有限流、降级或扩容预案。
```

### 6.2 生产告警建议

容量结论落地后，要转成告警。

建议告警项：

```text
1. 入口 RPS 超过生产安全水位；
2. P95 超过阈值；
3. P99 超过阈值；
4. 系统错误率超过阈值；
5. Hikari pending > 0 持续 N 分钟；
6. MySQL lock wait 增加；
7. Redis latency / slowlog 异常；
8. MQ 或本地队列堆积超过阈值；
9. 补偿任务堆积；
10. 库存、订单、流水一致性校验异常。
```

### 6.3 限流和降级策略

容量测试结论必须反推保护策略。

示例：

```text
系统有效容量：80 RPS
生产安全水位：50 RPS
限流启动阈值：50 RPS
强限流阈值：70 RPS
熔断阈值：P99 > 5s 或系统错误率 > 1%
降级策略：非核心查询走缓存兜底，送礼主链路返回稍后重试或排队中
扩容策略：应用实例从 2 扩到 4，验证 DB / Redis 是否仍可承载
```

## 8. 最小可执行套路

如果时间有限，至少执行以下流程：

```text
1. 定义 SLO：错误率、P95、P99、一致性；
2. 准备固定数据集；
3. 跑冒烟压测；
4. 跑分散写、热点 Gift、热点用户、双热点；
5. 对最关键场景做阶梯加压；
6. 找到容量拐点，取拐点前一档作为有效容量候选；
7. 用有效容量候选做 30 分钟长稳；
8. 做一次 3 倍突刺和恢复测试；
9. 执行一致性 SQL；
10. 输出有效容量、生产安全水位、瓶颈和保护策略。
```

最小结论格式：

```text
在【业务场景】下，系统有效容量为【X RPS】；
生产安全水位建议为【Y RPS】；
容量瓶颈为【具体资源或链路】；
超过安全水位后通过【限流/降级/扩容】保护；
一致性校验【通过/不通过】；
上线结论为【Go/No-Go】。
```
