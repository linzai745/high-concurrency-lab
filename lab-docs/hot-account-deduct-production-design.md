# 热点账户扣减生产级优化方案

## 1. 文档目标

本文档用于设计热点账户余额扣减的生产级优化方案。

这里的热点账户指：

```text
短时间内大量请求集中扣减同一个 user_account 账户余额，
导致数据库单行锁竞争、连接池等待、P95/P99 飙升、请求超时或系统错误。
```

本文只输出可生产落地的方案。以下方案不作为生产推荐：

```text
1. 单机固定 userId 分片队列；
2. userId % N 绑定固定单线程；
3. 内存队列无持久化、无背压、无恢复；
4. 只用 Redis DECR 扣余额，但没有账本、流水、冻结、回补、对账闭环；
5. 只把 DB 行锁等待迁移成应用内排队，却没有容量上限和保护策略。
```

***

## 2. 基本事实：单账户强一致扣减天然有串行上限

账户余额扣减不同于普通库存扣减。

余额属于资产账本，生产系统必须满足：

```text
1. 余额不能超扣；
2. 扣款不能丢；
3. 扣款不能重复；
4. 流水必须完整；
5. 失败必须可回滚或可补偿；
6. 每一笔余额变化都能审计；
7. 充值、消费、退款、冻结、解冻、补偿不能互相覆盖。
```

如果业务要求：

```text
同一个账户每次扣款都必须立即返回最终成功/失败，
并且余额不能为负，
并且所有扣款共享同一个可用余额。
```

那么同一个账户的扣减在逻辑上必须经过同一个余额约束点，无法无限并行。

单账户理论吞吐可以粗略估算：

```text
单账户最大稳定扣减 TPS ≈ 1000ms / 单次扣款事务 P95
```

示例：

```text
单次账户扣减事务 P95 = 20ms
单账户稳定上限 ≈ 50 TPS
```

如果业务希望同一个账户达到 200 TPS、500 TPS，则不能只优化 SQL 或线程池，必须改变业务模型：

```text
1. 限制单账户请求速率；
2. 合并多次扣款；
3. 接受异步确认；
4. 引入冻结/预占；
5. 将账户余额拆成多个可独立扣减的预算桶；
6. 设计完整账本和对账闭环。
```

***

## 3. 生产级方案选择原则

热点账户扣减不能先选技术方案，必须先确认业务语义。

### 3.1 必须先回答的问题

```text
1. 扣款接口是否必须同步返回最终成功/失败？
2. 是否允许返回“处理中”？
3. 是否允许短时间合并多次送礼？
4. 是否允许同一账户设置单用户限流？
5. 是否允许账户拆分为多个预算桶或子账户？
6. 是否存在退款、撤销、补偿、风控冻结？
7. 热点账户是普通用户账户，还是主播、商户、平台预算类账户？
8. 余额展示是否必须强实时？
9. 对账以账户余额为准，还是以账本流水为准？
10. 故障恢复后是否可以通过流水重建账户状态？
```

### 3.2 方案分层

生产上不要期望一个方案解决所有问题，应按层组合：

```text
第一层：正确性底座
DB 条件更新 + 账户流水 + 幂等 + 事务一致性。

第二层：流量保护
单账户限流、全局限流、超时、拒绝、熔断、降级。

第三层：降低写次数
请求合并、批量扣款、连击聚合。

第四层：改变执行模型
异步扣款、冻结/预占、账本确认。

第五层：拆分热点
预算桶、子账户、账户分片余额。
```

***

## 4. 推荐生产架构总览

默认推荐组合：

```text
1. 账户余额仍以 MySQL 账户表和账户流水为准；
2. 所有扣款必须有 requestId 幂等；
3. 扣款事务内同时完成余额条件更新和账户流水落库；
4. 对热点账户启用单账户限流和短窗口请求合并；
5. 对允许异步的场景，引入账户扣款命令表 / MQ，返回处理中；
6. 对极高频账户，使用预算桶或子账户拆分；
7. 所有异步、合并、拆分方案必须有对账、补偿、告警和回滚。
```

推荐优先级：

|优先级|方案|适用前提|解决的问题|
|---:|---|---|---|
|1|DB 条件更新 + 流水 + 幂等|所有账户扣减场景|正确性底座|
|2|单账户限流 + 背压|所有生产场景|防止热点账户拖垮系统|
|3|短窗口请求合并|连击送礼、批量打赏|减少 DB 更新次数|
|4|冻结/预占模型|订单可能失败、取消、退款|清晰表达资金占用状态|
|5|异步扣款命令模型|允许处理中状态|削峰和保护同步链路|
|6|预算桶 / 子账户拆分|超高频账户、平台类账户|提升单逻辑账户并发能力|

***

## 5. 方案一：正确性底座

### 5.1 目标

保证账户扣减在任何优化之前都是正确的。

### 5.2 核心表

账户表：

```text
user_account
- user_id
- available_balance
- frozen_balance
- version
- update_time
```

账户流水表：

```text
account_flow
- flow_no
- request_id
- user_id
- biz_type
- biz_no
- amount
- direction
- before_balance
- after_balance
- status
- create_time
```

幂等要求：

```text
account_flow.request_id 必须有唯一索引。
```

### 5.3 同步扣款 SQL

```sql
UPDATE user_account
SET available_balance = available_balance - #{amount},
    update_time = NOW()
WHERE user_id = #{userId}
  AND available_balance >= #{amount};
```

事务内必须同时完成：

```text
1. requestId 幂等校验；
2. 余额条件更新；
3. 账户流水写入；
4. 业务订单状态变更。
```

### 5.4 生产要求

```text
1. 事务范围只包含必要写操作；
2. SQL 必须命中 user_id 主键或唯一索引；
3. 不在事务内调用远程服务；
4. 不在事务内做耗时日志、网络 IO、复杂计算；
5. lock wait timeout、deadlock、duplicate key 必须分类处理；
6. 账户流水必须可用于对账和追溯。
```

该方案不能消除热点账户单行锁，但它是所有后续优化的正确性底座。

***

## 6. 方案二：单账户限流和背压

### 6.1 目标

防止单个热点账户把数据库连接池、应用线程和下游资源拖垮。

这是生产系统必须具备的保护层。

### 6.2 限流口径

按账户维度限流：

```text
key = account_deduct:{userId}
```

同时保留全局限流：

```text
key = account_deduct:global
```

限流阈值来自容量测试：

```text
单账户限流阈值 <= 单账户有效容量 * 安全系数
```

示例：

```text
单账户扣减事务 P95 = 20ms
单账户理论上限 ≈ 50 TPS
容量测试有效容量 = 35 TPS
安全系数 = 0.6
单账户生产限流阈值 ≈ 20 TPS
```

### 6.3 返回策略

超过阈值时不要无限排队。

可选返回：

```text
1. 请求过于频繁，请稍后重试；
2. 当前账户操作繁忙；
3. 已进入异步处理，稍后查询结果。
```

是否允许“处理中”取决于业务。

### 6.4 生产要求

```text
1. 限流必须支持多实例；
2. 限流必须有超时和失败兜底；
3. 限流失败时默认保护系统，而不是放开全部流量；
4. 限流命中率必须监控；
5. 单账户限流不能影响其他账户；
6. 不允许无限内存队列；
7. 拒绝请求必须有明确业务码。
```

***

## 7. 方案三：短窗口请求合并

### 7.1 适用场景

适用于用户连续送礼、连击打赏、批量扣款场景。

例如：

```text
同一用户 100ms 内连续送出 20 个相同礼物。
```

原始方式：

```text
20 次账户扣减
20 次账户流水
20 次订单写入
```

合并后：

```text
1 次账户扣减
1 笔聚合账户流水
20 条业务明细或 1 条聚合订单 + 明细
```

### 7.2 合并边界

只允许合并满足相同业务语义的请求：

```text
1. 同一个 userId；
2. 同一个 roomId；
3. 同一个 giftId；
4. 同一个价格版本；
5. 同一个支付账户；
6. 在固定短窗口内；
7. 每个原始 requestId 都要保留。
```

不满足条件不能合并。

### 7.3 数据模型

聚合扣款单：

```text
account_deduct_batch
- batch_no
- user_id
- total_amount
- request_count
- status
- create_time
```

聚合明细：

```text
account_deduct_batch_item
- batch_no
- request_id
- biz_no
- amount
- status
```

账户流水：

```text
account_flow.biz_no = batch_no
account_flow.amount = total_amount
```

### 7.4 生产要求

```text
1. 合并窗口必须有上限，通常 50ms ~ 200ms；
2. 合并不能改变用户可见语义；
3. 每个原始 requestId 仍然要幂等；
4. 失败时能定位到每个原始请求；
5. 聚合单和明细必须可对账；
6. 合并窗口过大，会增加用户感知延迟；
7. 合并必须有开关，可以按场景灰度。
```

### 7.5 效果

请求合并不能提升单次扣款 SQL 的速度，但能减少同一账户的写次数。

适用于：

```text
高频小额、短时间重复、业务允许聚合展示的场景。
```

不适用于：

```text
每笔请求必须独立实时确认且不能合并的支付场景。
```

***

## 8. 方案四：冻结 / 预占模型

### 8.1 目标

把“扣款”拆成更清晰的资金状态流转。

```text
可用余额 -> 冻结余额 -> 已消费
```

适用于订单可能失败、取消、退款、补偿的业务。

### 8.2 核心字段

```text
available_balance：可用余额
frozen_balance：冻结余额
```

预占：

```sql
UPDATE user_account
SET available_balance = available_balance - #{amount},
    frozen_balance = frozen_balance + #{amount},
    update_time = NOW()
WHERE user_id = #{userId}
  AND available_balance >= #{amount};
```

确认消费：

```sql
UPDATE user_account
SET frozen_balance = frozen_balance - #{amount},
    update_time = NOW()
WHERE user_id = #{userId}
  AND frozen_balance >= #{amount};
```

释放预占：

```sql
UPDATE user_account
SET available_balance = available_balance + #{amount},
    frozen_balance = frozen_balance - #{amount},
    update_time = NOW()
WHERE user_id = #{userId}
  AND frozen_balance >= #{amount};
```

### 8.3 生产要求

```text
1. 必须有 account_reservation 预占单；
2. 预占单 requestId 唯一；
3. 预占、确认、释放都必须写流水；
4. 预占超时必须自动释放；
5. 确认和释放必须幂等；
6. frozen_balance 不能长期异常堆积；
7. 必须有对账任务校验账户余额、冻结金额、流水。
```

### 8.4 注意事项

冻结 / 预占模型提升的是业务状态正确性和故障恢复能力。

它不能消除单账户行锁：

```text
同一个 user_account 仍然会被更新。
```

如果热点账户吞吐超过单行可承载能力，还需要结合限流、合并、异步或账户拆分。

***

## 9. 方案五：异步扣款命令模型

### 9.1 适用前提

只有当业务允许接口返回“处理中”时，才可以使用异步扣款。

如果接口必须同步返回最终扣款成功或失败，该方案不能作为主方案。

### 9.2 核心链路

```text
请求进入
↓
写 account_deduct_command 命令表，requestId 唯一
↓
通过 Outbox / MQ 投递扣款命令
↓
消费者执行 DB 条件扣款 + 写账户流水
↓
更新命令状态
↓
用户查询结果或通过消息通知
```

### 9.3 生产要求

```text
1. 必须使用命令表或 Outbox，不能只依赖内存队列；
2. MQ 消息必须可重复消费；
3. 消费端必须幂等；
4. 同一账户的命令处理要保证余额约束正确；
5. 队列必须有最大堆积阈值；
6. 超过堆积阈值必须拒绝或降级；
7. 必须监控生产速率、消费速率、消费延迟、堆积量；
8. 必须支持补偿扫描；
9. 用户侧必须有结果查询接口；
10. 必须定义处理中、成功、失败、超时的状态机。
```

### 9.4 热点隔离要求

异步模型不能简单依赖：

```text
accountId % partition
```

因为热点账户可能拖慢同一分区的其他账户。

生产上必须具备：

```text
1. 热点账户识别；
2. 热点账户单独限流；
3. 热点账户隔离队列或独立消费组；
4. 非热点账户不被热点账户阻塞；
5. 队列堆积后可快速降级或拒绝新请求。
```

### 9.5 适用场景

```text
1. 允许异步确认；
2. 高峰削峰；
3. 大促活动；
4. 用户可接受稍后查询结果；
5. 后台预算扣减、营销账户扣减。
```

## 10. 方案六：预算桶 / 子账户拆分

### 10.1 目标

当业务确实要求单个逻辑账户有很高并发扣减能力时，需要把一个逻辑账户拆成多个可独立扣减的余额桶。

示例：

```text
user_id = 10001
拆成 16 个 bucket：
10001#0
10001#1
...
10001#15
```

扣款时选择一个余额充足的 bucket。

### 10.2 适用账户

更适合：

```text
1. 平台营销预算账户；
2. 主播活动预算账户；
3. 商户余额账户；
4. 高频内部账户；
5. 明确会成为热点的特殊账户。
```

不建议默认用于所有普通用户账户。

### 10.3 数据模型

主账户：

```text
account
- account_id
- total_balance
- account_type
```

账户桶：

```text
account_bucket
- account_id
- bucket_no
- available_balance
- frozen_balance
- status
```

流水：

```text
account_flow
- flow_no
- account_id
- bucket_no
- request_id
- amount
- before_balance
- after_balance
```

### 10.4 扣款策略

可选策略：

```text
1. 按 requestId hash 到 bucket；
2. 选择余额充足的 bucket；
3. 多 bucket 组合扣款；
4. 后台定期 rebalance bucket 余额。
```

推荐先使用：

```text
hash 选择 bucket，失败后有限次尝试其他 bucket。
```

### 10.5 生产要求

```text
1. 总余额展示必须汇总所有 bucket；
2. 扣款流水必须记录 bucket_no；
3. 不允许因为 bucket 局部余额不足误判总余额不足；
4. 多 bucket 组合扣款必须有事务边界或补偿设计；
5. bucket 之间需要再平衡机制；
6. 退款必须回到原 bucket 或有明确规则；
7. 对账必须同时校验主账户、bucket、流水；
8. 该方案必须灰度到特定账户类型，不能盲目全量使用。
```

### 10.6 效果

预算桶 / 子账户拆分是少数能真正提升单逻辑账户并发扣减能力的方案。

代价是账务复杂度显著上升。

***

## 11. 不推荐方案：Redis 简单余额预扣

以下方案不作为生产推荐：

```text
Redis DECR user:balance:{userId}
异步 UPDATE MySQL
失败时 Redis INCR 回补
```

主要问题：

```text
1. 余额不是普通缓存，是资产状态；
2. Redis 成功但 MySQL 失败时语义复杂；
3. 订单成功、扣款成功、流水成功三者可能不一致；
4. 回补可能重复、遗漏或乱序；
5. 充值、退款、冻结、后台调整会和 Redis 余额冲突；
6. Redis 数据丢失或主从切换会影响资产状态；
7. 只靠 Redis 值无法解释每一分钱的变化。
```

如果要引入 Redis，只能作为：

```text
1. 限流计数；
2. 幂等短期缓存；
3. 热点识别；
4. 非权威余额展示缓存；
5. 配合预占单的快速校验，但不能替代账本和 DB 状态。
```

***

## 12. 推荐落地路径

### 12.1 第一阶段：先建立正确性和保护

```text
1. 保留 DB 条件更新；
2. 完善账户流水和 requestId 幂等；
3. 增加单账户限流；
4. 增加全局账户扣减限流；
5. 统计单账户 P95/P99 和有效容量；
6. 对热点账户返回明确拒绝或稍后重试。
```

目标：

```text
系统不会因为单个热点账户拖垮整体服务。
```

### 12.2 第二阶段：对高频送礼做请求合并

```text
1. 识别连击送礼场景；
2. 设计聚合订单和明细；
3. 短窗口合并扣款；
4. 保留原始 requestId 幂等；
5. 对账聚合流水和业务明细。
```

目标：

```text
减少同一账户的数据库更新次数。
```

### 12.3 第三阶段：引入冻结 / 预占

```text
1. 增加 available_balance / frozen_balance；
2. 增加 account_reservation；
3. 预占、确认、释放状态机；
4. 超时释放和补偿任务；
5. 对账任务。
```

目标：

```text
提升交易状态表达能力和故障恢复能力。
```

### 12.4 第四阶段：按业务选择异步或账户拆分

如果业务允许异步：

```text
使用扣款命令表 + Outbox / MQ + 状态查询。
```

如果业务必须提升单账户并发：

```text
只对明确热点账户类型启用预算桶 / 子账户拆分。
```

## 13. 容量测试方法

### 13.1 单账户容量测试

固定一个 userId：

```text
userId = 1
giftId 分散
每次扣款金额固定
逐步提升 RPS
```

记录：

```text

3. 系统错误率；
4. 余额不足率；
5. DB lock wait；
6. Hikari pending；
7. account_flow 写入延迟。
```

结论：

```text
得到单账户有效容量和生产限流阈值。
```

### 13.2 多热点账户容量测试

固定热点用户池：

```text
userId = 1 ~ 5
```

目标：

```text
验证多个热点账户同时存在时，
是否会拖垮数据库连接池和正常用户请求。
```

### 13.3 混合流量测试

示例：

```text
90% 普通用户
9% 中度热点用户
1% 极端热点用户
```

目标：


### 13.4 验收标准

```text
1. 普通账户请求 P95/P99 达标；
2. 热点账户超过阈值时被限流或进入异步；
3. 系统错误率达标；
4. Hikari pending 不持续增加；
5. DB lock wait 不扩散到全局；

7. 账户流水不丢、不重；

```

## 15. 方案选型表

|业务要求|推荐方案|
|---|---|
|必须同步返回最终结果，普通账户|DB 条件更新 + 流水 + 幂等 + 单账户限流|
|必须同步返回最终结果，高频连击|DB 条件更新 + 流水 + 幂等 + 单账户限流 + 请求合并|
|订单可能失败或取消|冻结 / 预占模型|
|允许返回处理中|扣款命令表 + Outbox / MQ + 状态查询|
|单逻辑账户必须支撑高并发|预算桶 / 子账户拆分|
|平台预算或商户账户|预算桶 / 子账户拆分 + 对账|
|普通用户异常高频|单账户限流 + 风控拦截|

***

## 16. 最终建议

热点账户扣减的生产级优化不能绕开账务正确性。

推荐落地组合：

```text
1. 用 DB 条件更新、账户流水、requestId 幂等做正确性底座；
2. 用单账户限流和全局限流保护系统；
3. 对连击送礼做短窗口合并，降低写次数；
4. 对需要状态流转的订单使用冻结 / 预占；
5. 对允许异步的场景使用命令表 + Outbox / MQ；
6. 对确实需要高并发的特殊账户使用预算桶 / 子账户拆分；
7. 全链路配套监控、告警、对账、补偿、灰度和回滚。
```

生产结论必须基于容量测试：

```text
单账户有效容量是多少；
单账户生产限流阈值是多少；
热点账户是否会影响普通账户；
异步或合并任务是否能追平；
账户余额、冻结金额、流水是否能对账通过。
```
