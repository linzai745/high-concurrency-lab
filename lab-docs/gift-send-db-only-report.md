# Gift 高并发送礼实验报告 - V1 DB Only 同步版
## 1. 实验背景
本实验用于模拟直播送礼场景下的高并发交易链路。

送礼链路本质上是一个典型的高并发写场景，请求进入后需要完成礼物校验、直播间校验、余额扣减、库存扣减、订单创建、账户流水记录等动作。

本版本为第一版基准实验，``采用 DB Only 同步事务模式``，所有核心操作都直接访问 MySQL，不引入 Redis 缓存、Redis 预扣库存、MQ 削峰、限流降级等优化手段。

本版本的目的不是追求最高吞吐，而是为了建立基准数据，并暴露同步数据库链路在高并发下的主要瓶颈。
***
## 2. 实验目标
本次实验主要验证一下问题：
```text
1. 同步数据库版本的基础吞吐能力
2. 高并发下库存扣减是否会出现超卖
3. 高并发下余额扣减是否会出现超扣
4. 幂等 requestId 是否能防止重复订单
5. 热点 Gift 库存行是否会产生锁竞争
6. 少量用户高频送礼时，账户余额行是否会产生锁竞争
7. 订单表和账户流水表持续写入对 RT 的影响
8. MySQL 连接池、CPU、慢 SQL、行锁等待是否成为瓶颈
```

## 5. 数据库一致性策略
### 5.1 余额防超扣
余额扣减使用条件更新：
```sql
UPDATE user_account
SET balance = balance - #{amount},
    update_time = NOW()
WHERE user_id = #{userId}
  AND balance >= #{amount};
```
这个设计依赖 MySQL InnoDB 行锁和条件更新保证不会出现余额扣成负数。

当余额不足时，SQL 影响行数为 0， 业务返回余额不足。
***
### 5.2 库存防超卖
库存扣减使用条件更新：
```SQL
UPDATE gift
SET stock = stock - #{count},
    update_time = NOW()
WHERE id = #{giftId}
  AND stock >= #{count}
  AND status = 1;
```
这个设计同样依赖 MySQL InnoDB 行锁和条件更新保证不会出现库存扣成负数。

当库存不足时，SQL 影响行数为0，业务返回库存不足。
***
### 5.3 请求幂等
当前版本通过 gift_order.request_id 唯一索引做幂等兜底：
```text
UNIQUE KEY uk_request_id (request_id)
```
应用层先根据 requestId 查询订单：
```text
如果订单已存在，直接返回已有 orderNo
如果订单不存在，继续执行送礼流程
```
需要注意：当前版本的幂等是“先查再写”模式。在完全并发的相同 requestId 请求下，可能多个线程同时查不到数据，最终需要依赖数据库唯一索引兜底。

后续版本可以引入 Redis SETNX 或专门的幂等组件做强前置幂等控制。

## 11. 现象分析
### 11.1 会随着并发上升明显增加
原因是当前所有请求都进入 MySQL，同步完成余额扣减、库存扣减、订单插入和流水插入。

一次请求至少包含：
```text
1 次 gift_order 幂等查询
1 次 gift 查询
1 次 live_room 查询
1 次 user_account 余额扣减
1 次 gift 库存扣减
1 次 gift_order 插入
1 次 account_flow 插入
```
也就是说，单次送礼请求会产生多次数据库访问。

当并发升高后，数据库连接池、MySQL CPU、行锁等待都会影响接口 RT。
***
### 11.2 热点 Gift 会导致库存行锁竞争
所有请求都扣减同一个giftId = 1 的库存。

核心SQL：
```text
UPDATE gift
SET stock = stock - #{count}
WHERE id = #{giftId}
  AND stock >= #{count}
  AND status = 1;
```
由于所有请求都更新同一行数据，InnoDB 会对该行加行锁。并发越高，等待同一行锁的请求越多。

所以这个版本能保证不超卖，但吞吐会受到热点行锁限制。
***
### 11.3 少量用户会导致余额行锁竞争
当前用户只从 userId = 1 ~ 5 中随机选择。

这意味着大量请求会集中扣减少数几行账户余额。

核心 SQL：
```text
UPDATE user_account
SET balance = balance - #{amount}
WHERE user_id = #{userId}
  AND balance >= #{amount};
```
这同样会导致账户余额行锁竞争。

如果压测用户数量扩大到 1 万个，账户行锁竞争会下降，但 Gift 库存行仍然是热点。
***
### 11.4 订单表和流水表会带来持续写入压力
每次成功送礼都会插入：
```text
gift_order 1 条
account_flow 1 条
```
所以成功请求越多，数据库写入压力越高。
当 TPS 上升时，订单表和流水表的写入会影响整体吞吐。
***
### 11.5 当前幂等方式不适合极端并发
当前幂等逻辑是：
```text
先查 gift_order.request_id
再执行业务逻辑
最后插入 gift_order
```
这种方式在普通重复请求下可以正常返回已有订单。

但是在相同 requestId 高并发同时进入时，多个线程可能同时查不到订单，继续往下执行。最后只有一个线程插入订单成功，其他线程会被数据库唯一索引拦住。

这个设计有数据库兜底，但不够优雅。后续需要前置幂等组件，避免重复请求进入核心扣减链路。
***
## 12. 瓶颈判断
当前版本的主要瓶颈可以归纳为：

| 瓶颈点        | 表现           | 原因                     |
|------------|--------------|------------------------|
| Gift 库存热点行 | P95/P99 升高   | 所有请求更新同一行库存            |
| 用户余额热点行    | RT 波动        | 少量用户反复扣减余额             |
| 数据库连接池     | 请求等待连接       | 所有读写都走 MySQL           |
| 订单流水写入     | TPS 上升后写压力增加 | 每次请求同步写两张表             |
| 幂等查询       | 增加一次 DB 读    | requestId 先查后写         |
| 长事务        | 锁持有时间变长      | 扣余额、扣库存、写订单、写流水都在一个事务中 |
***
## 13. 当前版本结论
本版本采用 DB Only 同步事务模式，实现了 Gift 送礼链路的基础正确性。

当前版本可以保证：
```text
1. 不会出现库存超卖
2. 不会出现余额超扣
3. 成功请求会生成订单
4. 成功请求会生成账户流水
5. requestId 唯一索引可以兜底防止重复订单
```
但当前版本在高并发下存在明显性能瓶颈：
```text
1. 热点 Gift 库存扣减依赖 MySQL 行锁，吞吐受限
2. 少量用户高频送礼时，账户余额行会产生锁竞争
3. 所有请求都直接访问 MySQL，数据库压力较大
4. 所有动作都在一个事务中，锁持有时间较长
5. 幂等逻辑没有强前置，极端并发下仍会让重复请求进入主链路
```
因此，V1 版本适合作为后续优化实验的基准版本。

## V1 对照实验：热点写场景 vs 分散写场景
### 1. 对照实验目的
在第一轮压测中，V1 DB Only 同步事务版出现了大量 500 错误，主要异常包括：
```text
CannotCreateTransactionException
Connection is not available
HikariPool connection timeout
```
同时在 MySQL 侧观察到：
```text
processlist 中存在大量 updating 连接
InnoDB 状态中存在大量 lock wait
```
初步判断是高并发请求集中更新热点资源，导致数据库行锁等待，进而引发事务堆积和 Hikari 连接池耗尽。

为了验证这个判断，本轮增加一个对照实验：将请求从少量用户、单个礼物、调整为大量用户、多个礼物，使写入压力分散到更多数据库行上。
***
### 2. 压测场景对比

### 4. V1-A 热点写场景分析
在 V1-A 场景中，请求集中在极少数数据库行上：
```text
100 VU
    ↓
5 个 userId
    ↓
1 个 giftId
```
核心更新 SQL 包括：
```text
UPDATE gift
SET stock = stock - #{count}
WHERE id = #{giftId}
  AND stock >= #{count}
  AND status = 1;
```
以及：
```text
UPDATE user_account
SET balance = balance - #{amount},
    update_time = NOW()
WHERE user_id = #{userId}
  AND balance >= #{amount};
```
由于所有请求都集中扣减同一个 giftId = 1的库存，MySQL InnoDB 会对该库存行进行加排他锁。其他并发事务只能等待该行锁释放。
同时，用户只分布在 userId = 1 ~ 5，账户余额扣减也会集中在少量行上，进一步加剧锁等待。

最终形成如下链路：
```text
热点库存行 / 热点账户行竞争
        ↓
大量事务处于 lock wait
        ↓
数据库连接长时间被事务占用
        ↓
Hikari active 连接被打满
        ↓
后续请求无法获取数据库连接
        ↓
CannotCreateTransactionException
        ↓
接口返回 500
```
因此，V1-A 场景中出现 76.91% 失败率，并不是因为业务代码无法执行，而是 DB Only 同步事务模式在热点写场景下出现了典型的：
```text
行锁等待 → 事务堆积 → 连接池耗尽 → 大量请求失败
```
***
### 5. V1-B 分散写场景分析
在 V1-B 场景中，请求分布被调整为：
```text
100 VU
    ↓
1000 个 userId
    ↓
100 个 giftId
```
写压力被分散到更多数据库行上：
```text
100 行 gift 库存
1000 行 user_account 余额
```
因此，单行锁竞争明显下降。事务可以更快完成，数据库连接也可以更快归还给 Hikari 连接池。

压测结果显示：
```text
总请求数：3188
成功请求数：3188
失败请求数：0
成功率：100%
失败率：0%
RPS：51.65/s
平均 RT：1.9s
P95：3.71s
```
相比 V1-A：
```text
失败率从 76.91% 降到 0%
P95 从 17.12s 降到 3.71s
RPS 从 15.67/s 提升到 51.65/s
```
这说明，上一轮大量 500 的核心原因确实是热点行锁竞争，而不是 k6 压测工具问题，也不是接口基本功能错误。

### 6. RPS 与 RT 的关系说明
在 100 VU 下，理论吞吐和平均响应时间大致满足：
```text
RPS ≈ 并发数 / 平均 RT
```
V1-B 场景中：
```text
并发数 = 100
平均 RT = 1.9s
```
理论 RPS 约为：
```text
100 / 1.9 ≈ 52.63 RPS
```
实际压测结果为：
```text
51.65 RPS
```
二者基本一致，说明当前吞吐主要受服务端响应时间限制。

换句话说，并不是 k6 没有打出压力，而是服务端平均每个请求需要处理的 1.9 秒，因此 100 个 VU 下只能完成每秒 50 次左右的请求。
***
### 7. 对照实验结论
通过 V1-A 和 V1-B 两组压测可以得出以下结论。
#### 7.1 热点行锁是 V1-A 大量 500 的核心原因
V1-A 中，请求集中更新同一个 Gift 库存行和少量账户余额行，导致 InnoDB 出现大量 lock wait，应用侧出现 Hikari 连接池耗尽，最终触发大量 500。

V1-B 中，将请求分散到 100 个礼物和 1000 个用户后，失败率从 76.91% 降为 0%，证明热点行锁竞争是 V1-A 的核心问题。
***
#### 7.2 DB Only 同步事务版可以承载分散写，但不适合热点写
在资源分散的情况下，DB Only 同步事务版可以保持数据正确性和请求成功率。

但是在热点资源集中更新场景下，数据库行锁会把并发请求串行化，导致事务堆积、连接池耗尽和接口失败。

因此，DB Only 同步事务模式更适合低热点、写入分散的业务场景，不适合作为热点 Gift、热点库存、秒杀库存这类高并发写热点场景的最终方案。
***
#### 7.3 当前版本仍然存在单链路过重问题
虽然 V1-B 成功率达到 100%，但平均 RT 仍然达到 1.9 秒，P95 仍然达到 3.71 秒。

这说明即使热点被打散，当前链路仍然偏重。

一次送礼请求仍然包含多次数据库操作：
```text
requestId 幂等查询
Gift 查询
LiveRoom 查询
用户余额扣减
Gift 库存扣减
GiftOrder 插入
AccountFlow 插入
事务提交
```

因此，V1-B 解决的是稳定性问题，但没有彻底解决性能问题。
***
### 8. 当前版本最终结论
V1 DB Only 同步事务版具备基础正确性，在正常分散写场景下可以完成送礼链路，并保证余额不超扣、库存不超卖、订单和流水一致写入。

但是，本次对照实验也证明：
```text
1. 当请求集中在单个 giftId 和少量 userId 上时，MySQL 行锁竞争会迅速放大；
2. 行锁等待会导致事务长期占用数据库连接；
3. 数据库连接无法及时归还，会进一步导致 Hikari 连接池耗尽；
4. 连接池耗尽后，应用无法创建事务，最终产生大量 500；
5. 将 userId 和 giftId 打散后，失败率可以从 76.91% 降到 0%，说明热点写竞争是主要瓶颈；
6. 即使写热点被打散，平均 RT 仍有 1.9 秒，说明同步数据库链路本身仍然较重。
```
因此，V1 版本适合作为后续优化实验的基准版本。
***
## 结论
```text
V1 DB Only 同步事务版的核心价值是建立基准数据，并暴露同步数据库链路在不同写入分布下的表现。

在 5 个用户、1 个礼物的热点写场景中，系统出现 76.91% 的失败率，P95 达到 17.12 秒。结合应用日志和 MySQL 状态可以确认，主要原因是热点库存行和少量账户行产生严重 InnoDB 行锁等待，导致事务长期占用数据库连接，最终引发 Hikari 连接池耗尽和大量 500。

在调整为 1000 个用户、100 个礼物后，失败率降为 0%，RPS 从 15.67/s 提升到 51.65/s，P95 从 17.12 秒下降到 3.71 秒。这证明上一轮的问题主要来自热点行锁竞争，而不是接口功能错误。

因此，DB Only 同步事务版在写入分散场景下可以保持正确性和稳定性，但不适合热点库存、热点账户等高并发写热点场景。后续需要通过缓存前置降低读路径压力，通过 Redis 预扣库存、库存分桶、MQ 削峰和限流背压等方式解决热点写竞争和系统保护问题。
```
