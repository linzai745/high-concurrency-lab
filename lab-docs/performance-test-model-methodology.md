# 高并发压测模型方法论

## 1. 文档目标

本文档用于规范 Gift 高并发送礼系统的压测模型设计方法。

它解决的问题不是“压测时怎么排查故障”，而是：

```text
1. 压测前应该如何建模；
2. 应该设计哪些压测场景；
3. 如何设置并发、数据分布、持续时间和阈值；
4. 如何判断一次压测结论是否可信；
5. 如何通过多轮压测找到系统容量边界。
```

压测不是简单把 VU 调大。可落地的压测必须回答：

```text
在什么业务场景、什么数据分布、什么流量模型、什么成功标准下，系统能稳定支撑多少吞吐。
```

***

## 2. 压测建模的核心原则

### 2.1 先建业务模型，再建技术模型

压测模型必须先描述业务事实，再转换成技术参数。

业务事实包括：

```text
1. 谁在请求：用户数量、用户分布、热点用户比例；
2. 请求什么：接口、礼物、直播间、库存、账户；
3. 请求频率：平均流量、峰值流量、突发流量；
4. 请求结果：成功、库存不足、余额不足、重复请求；
5. 一致性要求：余额不能超扣、库存不能超卖、订单不能重复；
6. 用户体验要求：P95、P99、超时阈值。
```

技术模型再落到：

```text
1. k6 executor；
2. VU 数；
3. duration；
4. arrival rate；
5. 数据池大小；
6. 热点比例；
7. 阈值 thresholds；
8. 观测指标。
```

### 2.2 压测结论必须绑定场景

不能只写：

```text
系统支持 100 QPS。
```

必须写成：

```text
在 1000 用户、100 礼物、100 VU、持续 60 秒、错误率 0%、P95 3s 以内的分散写场景下，系统稳定支撑约 60 RPS。
```

同一套系统在不同数据分布下容量会完全不同。

例如：

```text
分散用户 + 分散礼物：更接近普通场景；
分散用户 + 热点礼物：验证库存热点；
热点用户 + 分散礼物：验证账户热点；
热点用户 + 热点礼物：验证极端双热点。
```

### 2.3 不用平均值定义容量

压测通过标准必须包含：

```text
1. 错误率；
2. P95；
3. P99；
4. RPS / TPS；
5. 一致性校验结果。
```

平均 RT 只能作为辅助指标。

推荐表达：

```text
当错误率 < 1%，P95 < 3s，P99 < 5s，且库存、余额、订单一致性校验通过时，本轮流量视为可稳定承载。
```

## 3. 压测模型五要素

一次完整压测必须明确五个模型。

### 3.1 流量模型

流量模型回答：

```text
请求以什么节奏进入系统。
```

常见模型：

|模型|说明|适用场景|
|---|---|---|
|固定并发模型|固定 VU 持续请求|观察系统在固定压力下的稳定性|
|阶梯加压模型|逐步增加 VU 或 RPS|寻找容量拐点|
|固定到达率模型|固定每秒请求数|模拟真实流量入口|
|突刺模型|短时间突然放大流量|验证削峰、限流、队列|
|长稳模型|较长时间稳定压测|验证内存、连接、任务堆积和一致性|

本项目建议：

```text
1. 功能基准：固定并发模型；
2. 容量摸底：阶梯加压模型；
3. 稳定性验证：长稳模型；
4. 削峰验证：突刺模型；
5. 生产容量评估：固定到达率模型。
```

### 3.2 数据模型

数据模型回答：

```text
请求打到哪些用户、礼物、直播间、库存和账户。
```

必须定义：

```text
1. userId 范围；
2. giftId 范围；
3. roomId 范围；
4. 每个用户初始余额；
5. 每个礼物初始库存；
6. 每次送礼数量；
7. requestId 生成规则；
8. 是否允许业务失败。
```

错误示例：

```text
随机 userId、随机 giftId，没有记录数据范围，也没有压测后校验。
```

正确示例：

```text
userId = 1 ~ 1000
giftId = 1 ~ 100
roomId = 20001
每个用户 balance = 1000000
每个礼物 stock = 1000000
每次送 1 个礼物
requestId = 场景名 + VU + iteration + timestamp
压测后校验订单数、余额扣减、库存扣减、预扣流水。
```

### 3.3 热点模型

热点模型回答：

```text
请求是否集中打到少量数据行。
```

本项目必须固定四类热点模型：

|模型|用户分布|礼物分布|验证目标|
|---|---|---|---|
|分散写|1000 用户|100 礼物|普通交易链路吞吐|
|热点 Gift|1000 用户|1 礼物|库存热点和 gift.stock 行锁|
|热点用户|5 用户|100 礼物|账户热点和 user_account 行锁|
|双热点|5 用户|1 礼物|极端排队、超时和降级能力|

热点比例也可以进一步细化：

```text
80/20 模型：80% 请求打到 20% 热点数据；
95/5 模型：95% 请求打到 5% 热点数据；
100/1 模型：所有请求打到同一个热点 Gift 或热点用户。
```

建议演进顺序：

```text
1. 先跑 100/1 极端热点，暴露瓶颈；
2. 再跑 80/20 或 95/5，模拟真实热点；
3. 最后跑分散写，确认优化没有伤害普通场景。
```

### 3.4 成功标准模型

成功标准回答：

```text
什么情况下认为系统“扛住了”。
```

建议分三层定义。

第一层：接口可用性

```text
系统错误率 < 1%
HTTP 5xx = 0 或接近 0
连接失败 = 0
超时率 < 1%
```

第二层：性能体验

```text
P95 < 3000ms
P99 < 5000ms
平均 RT 仅作为参考
RPS 在压测中后段保持稳定
```

第三层：业务一致性

```text
订单不重复；
余额不超扣；
库存不超卖；
Redis 库存和 MySQL 库存最终可对齐；
库存预扣流水状态可闭环；
失败请求有明确业务原因或系统原因。
```

不同阶段阈值可以不同，但必须在压测前写清楚。

### 3.5 观测模型

观测模型回答：

```text
压测过程中要看哪些指标，压测后要保留哪些证据。
```

最低要求：

```text
1. k6 输出；
2. 应用日志；
3. Actuator metrics；
4. MySQL processlist / lock wait；
5. Redis INFO / SLOWLOG；
6. JVM GC 和线程栈；
7. 压测后数据一致性 SQL。
```

没有观测模型的压测只能得到现象，不能得到结论。

## 4. 推荐压测阶段

### 4.1 阶段一：单接口冒烟压测

目标：

```text
确认接口、数据、脚本、返回判断都正确。
```

建议参数：

```text
VU：1 ~ 5
duration：30s
数据：少量用户、少量礼物
```

通过标准：

```text
1. 请求全部成功或业务失败符合预期；
2. requestId 没有重复；
3. 订单、余额、库存数据正确；
4. k6 checks 能正确区分成功和失败。
```

### 4.2 阶段二：基准压测

目标：

```text
得到当前版本在标准场景下的基础吞吐和延迟。
```

建议参数：

```text
VU：50 或 100
duration：60s
场景：分散写、热点 Gift、热点用户、双热点
```

输出：

```text
1. 每个场景的 RPS；
2. 平均 RT、P90、P95、P99；
3. 错误率；
4. 主要瓶颈判断；
5. 一致性校验结果。
```

### 4.3 阶段三：阶梯加压

目标：

```text
找到系统容量拐点。
```

建议加压方式：

```text
VU：20 -> 50 -> 100 -> 200 -> 400
每档持续：2 ~ 5 分钟
每档之间保留观察窗口
```

判断容量拐点：

```text
1. VU 增加但 RPS 不再明显上升；
2. P95 / P99 开始快速上升；
3. 错误率开始出现；
4. 连接池、线程池、DB 锁或队列开始堆积。
```

容量结论写法：

```text
在某场景下，系统在 100 VU 时 RPS 约 60/s，P95 约 3s，错误率 0%；
升到 200 VU 后 RPS 未增长，P95 上升到 8s，出现连接池等待；
因此当前场景有效容量上限在 100 VU 附近，主要瓶颈为 MySQL 行锁等待。
```

### 4.4 阶段四：长稳压测

目标：

```text
验证系统在稳定流量下是否存在资源泄漏、队列堆积、补偿堆积、数据不一致。
```

建议参数：

```text
duration：30 分钟、1 小时或更长
压力：选择低于容量拐点 60% ~ 70% 的流量
场景：生产最常见场景 + 一个热点场景
```

重点观察：

```text
1. P95 / P99 是否随时间上升；
2. 内存是否持续增长；
3. Hikari active / pending 是否持续上升；
4. Redis 库存是否正常扣减；
5. gift_stock_reservation 是否堆积；
6. stock-sync / recover / compensate 是否能追平；
7. 余额、库存、订单是否一致。
```

### 4.5 阶段五：突刺压测

目标：

```text
验证系统面对短时流量洪峰时的保护能力。
```

适用能力：

```text
1. Redis 库存预扣；
2. 本地队列；
3. 账户分片队列；
4. MQ 削峰；
5. 限流和降级。
```

建议模型：

```text
正常流量 1 分钟
突增到 3 ~ 5 倍流量 30 秒
回落到正常流量 2 分钟
```

重点看：

```text
1. 突刺时错误率是否可控；
2. 队列是否溢出；
3. P99 是否不可接受；
4. 突刺结束后系统是否恢复；
5. 异步任务和补偿任务是否能追平。
```

## 5. 本项目标准压测场景矩阵

### 5.1 基础矩阵

|场景|用户数量|礼物数量|直播间数量|目标|
|---|---:|---:|---:|---|
|分散写|1000|100|1|验证普通吞吐|
|热点 Gift|1000|1|1|验证库存热点|
|热点用户|5|100|1|验证账户热点|
|双热点|5|1|1|验证极端冲突|

每个版本都至少跑这四组。

### 5.2 版本验证重点

|版本|主要目标|必须跑的场景|重点指标|
|---|---|---|---|
|V1 DB Only|建立数据库同步事务基准|四组基础矩阵|DB 行锁、Hikari、P95/P99|
|V2 Cache Preload|验证读缓存收益|分散写、热点 Gift|DB 读压力、P95 变化|
|V3 Redis Stock|验证库存热点移除|热点 Gift、双热点|Redis 预扣、gift 行锁是否消失|
|V4 Account Guard|验证账户热点保护|热点用户、双热点|账户队列、user_account 锁等待|
|V5 MQ|验证削峰异步化|突刺、长稳|队列堆积、消费延迟、最终一致性|

### 5.3 每组场景都必须输出

```text
1. 测试配置；
2. k6 结果；
3. 错误分类；
4. 容量拐点；
5. 主要瓶颈；
6. 一致性校验；
7. 和上一版本对比；
8. 下一步优化动作。
```

## 6. k6 脚本设计规范

### 6.1 requestId 规范

requestId 必须全局唯一，且可追踪来源。

推荐格式：

```text
{scenario}-{vu}-{iteration}-{timestamp}-{random}
```

禁止：

```text
1. 使用固定 requestId；
2. 只用 userId + giftId；
3. 多个场景复用同一批 requestId；
4. 压测失败后不区分重试请求和新请求。
```

### 6.2 checks 规范

k6 checks 至少区分：

```text
1. HTTP 状态码是否符合预期；
2. 业务 code 是否成功；
3. 库存不足是否属于预期；
4. 余额不足是否属于预期；
5. 系统异常是否单独统计。
```

不要只用 HTTP 200 判断成功。

### 6.3 thresholds 规范

示例：

```javascript
export const options = {
  thresholds: {
    http_req_failed: ['rate<0.01'],
    http_req_duration: ['p(95)<3000', 'p(99)<5000'],
  },
};
```

热点实验可以放宽阈值，但必须说明原因：

```text
热点写实验允许更高失败率，是为了暴露瓶颈；
稳定性验收不能放宽系统错误率。
```

### 6.4 数据分布规范

脚本中必须显式表达数据分布。

示例：

```javascript
const USER_COUNT = 1000;
const GIFT_COUNT = 100;

const userId = 1 + Math.floor(Math.random() * USER_COUNT);
const giftId = 1 + Math.floor(Math.random() * GIFT_COUNT);
```

热点 Gift：

```javascript
const giftId = 1;
```

热点用户：

```javascript
const HOT_USERS = [1, 2, 3, 4, 5];
const userId = HOT_USERS[Math.floor(Math.random() * HOT_USERS.length)];
```

## 7. 数据一致性校验模型

性能压测必须同时做一致性校验。

### 7.1 订单幂等校验

```sql
SELECT request_id, COUNT(*) AS cnt
FROM gift_order
GROUP BY request_id
HAVING cnt > 1;
```

期望：

```text
返回 0 行。
```

### 7.2 余额不超扣校验

```sql
SELECT user_id, balance
FROM user_account
WHERE balance < 0;
```

期望：

```text
返回 0 行。
```

### 7.3 库存不超卖校验

```sql
SELECT id, stock
FROM gift
WHERE stock < 0;
```

期望：

```text
返回 0 行。
```

### 7.4 订单和流水数量校验

```sql
SELECT COUNT(*) FROM gift_order;
SELECT COUNT(*) FROM account_flow;
```

期望：

```text
成功订单数和账户流水数口径一致。
```

### 7.5 Redis 预扣流水校验

```sql
SELECT status, COUNT(*)
FROM gift_stock_reservation
GROUP BY status;
```

期望：

```text
待同步、同步中、待补偿状态不能长期堆积。
```

## 8. 容量评估方法

### 8.1 有效容量定义

有效容量不是最大瞬时 RPS，而是满足约束的稳定吞吐。

定义：

```text
有效容量 = 在指定业务场景下，错误率、P95、P99、一致性都满足阈值时的最大稳定 RPS。
```

示例：

```text
场景：分散写
错误率阈值：< 1%
P95 阈值：< 3000ms
P99 阈值：< 5000ms
一致性：全部通过
结论：有效容量约 60 RPS
```

### 8.2 容量拐点识别

阶梯加压中，当出现以下情况，说明已经越过容量拐点：

```text
1. 压力增加，RPS 不再上升；
2. P95 / P99 成倍上升；
3. 错误率开始出现；
4. 连接池 pending 增加；
5. DB 锁等待增加；
6. 队列长度持续增加。
```

容量应该取拐点前一档，而不是崩溃那一档。

### 8.3 安全水位

生产容量不能按实验室极限值直接使用。

建议：

```text
生产建议水位 = 有效容量的 50% ~ 70%
```

如果系统依赖异步削峰、补偿任务或最终一致性链路，还需要额外保留：

```text
1. 队列堆积空间；
2. 消费追平能力；
3. DB 连接余量；
4. Redis 余量；
5. 故障恢复余量。
```

## 10. 最小落地流程

一次合格压测至少按以下流程执行：

```text
1. 写清楚业务场景和成功标准；
2. 准备固定测试数据；
3. 执行单接口冒烟压测；
4. 执行四组基础场景；
5. 对关键场景做阶梯加压；
6. 对目标容量做长稳压测；
7. 保存 k6、应用、MySQL、Redis、JVM 证据；
8. 执行一致性 SQL；
9. 写出容量、瓶颈和下一步动作；
10. 和上一版本做同场景对比。
```

压测结论必须做到：

```text
有场景、有阈值、有数据、有证据、有校验、有下一步。
```
