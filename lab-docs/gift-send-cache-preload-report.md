# Gift 高并发送礼实验报告 - V2 Cache Preload 缓存前置版

## 1. 实验背景

V1 DB Only 同步版已经证明了两个问题：
```text
1. 在分散写场景下，DB Only 可以保持稳定，但 RT 偏高；
2. 在热点写场景下，单 Gift 库存行和少量账户行会产生严重行锁竞争，导致事务堆积、连接池耗尽和大量请求失败。
```
V2 在不改变核心写入一致性策略的前提下，引入 Gift / LiveRoom 读路径缓存前置，用于减少每次送礼请求中的 MySQL 读查询。

V2 版本仍然保留：
```
余额扣减：MySQL 条件更新
库存扣减：MySQL 条件更新
订单创建：MySQL 同步写入
账户流水：MySQL 同步写入
```
也就是说，V2 不是库存热点写优化版本，而是读路径优化版本。
***


## 2. V2 优化内容

V1 请求链路：
```
查订单 DB
查 Gift DB
查 LiveRoom DB
扣余额 DB
扣库存 DB
写订单 DB
写流水 DB
```
V2 请求链路：
```
查订单 DB
查 Gift Cache
查 LiveRoom Cache
扣余额 DB
扣库存 DB
写订单 DB
写流水 DB
```
V2 的核心变化：
```
1. Gift 基础信息走本地缓存；
2. LiveRoom 状态走本地缓存；
3. 真实余额扣减仍然走 MySQL；
4. 真实库存扣减仍然走 MySQL；
5. 不使用缓存库存参与扣减判断；
6. 不改变订单和流水的同步写入方式。
```
因此，V2 的目标是：
```
降低读路径 DB 压力，观察平均 RT、P95 和 RPS 是否改善。
```

## 6. RPS 与 RT 关系分析

### 6.1 分散写场景

V2 分散写场景：
```
并发数：100 VU
平均 RT：1.66s
```
理论 RPS 约为：
```
100 / 1.66 ≈ 60.24 RPS
```
实际 RPS：
```
59.38 RPS
```
两者非常接近，说明当前吞吐主要受服务端平均响应时间限制。
***

### 6.2 热点写场景

V2 热点写场景：
```
并发数：100 VU
平均 RT：19.86s
```
理论 RPS 约为：
```
100 / 19.86 ≈ 5.03 RPS
```
实际 RPS：
```
4.26 RPS
```
两者也接近，说明热点写场景下吞吐严重受请求等待时间影响。
***

## 7. V2 分散写场景结论

V2 在分散写场景下是有效的。

原因是分散写场景中，giftId 和 userId 都被打散：
```
1000 个 userId
100 个 giftId
```
单行锁竞争较低，因此减少 Gift / LiveRoom 的 MySQL 查询后，可以直接改善整体链路耗时。

V2 分散写场景证明：
```
缓存前置对读多写少数据有效；
在低热点写模型下，缓存可以降低平均 RT 和 P95，并提升 RPS。
```
但是，V2 分散写场景仍然没有达到 3s 的 P95 阈值：
```
P95 = 3.23s
```
说明当前主链路仍然存在性能瓶颈，包括：
```
1. requestId 幂等查询仍然走 DB；
2. 余额扣减仍然走 DB；
3. 库存扣减仍然走 DB；
4. 订单写入仍然同步；
5. 账户流水写入仍然同步；
6. 整体事务链路仍然偏重。
```
***

## 8. V2 热点写场景结论

V2 在热点写场景下只能缓解失败率，不能解决性能问题。

热点写场景中，请求集中在：
```
5 个 userId
1 个 giftId
```
核心瓶颈仍然是：
```
1. gift.id = 1 的库存行锁竞争；
2. user_account.user_id = 1 ~ 5 的账户余额行锁竞争。
```
即使 Gift 和 LiveRoom 查询走了缓存，请求最终仍然要执行：
```
UPDATE gift
SET stock = stock - #{count}
WHERE id = #{giftId}
AND stock >= #{count}
AND status = 1;
```
以及：
```
UPDATE user_account
SET balance = balance - #{amount}
WHERE user_id = #{userId}
AND balance >= #{amount};
```
因此，高并发请求仍然会在 MySQL 写锁上串行排队。

本轮热点写结果说明：
```
V2 没有根治热点写锁；
缓存前置无法解决库存单行更新竞争；
热点写场景需要进入 V3 写路径优化。
```

## 10. V2 核心结论

V2 Cache Preload 缓存前置版的实验结论如下：
```
1. V2 在分散写场景下有效。
   RPS 从 51.65/s 提升到 59.38/s，平均 RT 从 1.90s 下降到 1.66s，P95 从 3.71s 下降到 3.23s。
2. V2 在分散写场景下保持 0% 失败率。
   说明 Gift / LiveRoom 缓存前置不会破坏余额扣减、库存扣减、订单创建和流水写入的一致性。
3. V2 在热点写场景下没有根治问题。
   虽然失败率从 76.91% 降到 12.91%，但平均 RT 从 5.43s 上升到 19.86s，P95 从 17.12s 上升到 28.46s。
4. 热点写场景下，V2 的表现说明请求不再大量快速失败，而是更多请求进入 MySQL 行锁等待队列，最终导致响应时间极长。
5. 缓存前置只能优化读路径，不能解决库存扣减和账户扣减这类写热点竞争。
6. 当前系统下一步优化重点应从读路径转向写路径。
```
一句话总结：
```
V2 证明了缓存前置对分散写场景有效，但也证明了热点写问题的根因不在读查询，而在 MySQL 写锁竞争。
```

## 11. 为什么热点写 V2 失败率下降但 RT 上升？

这个现象非常重要。

V1 热点写时：
```
大量请求在事务创建阶段拿不到连接
很快抛出 CannotCreateTransactionException
返回 500
```
所以 V1 有很多失败请求是“快速失败”。

V2 热点写时：
```
Gift / LiveRoom 读查询减少
事务范围收窄
连接池压力有所缓解
更多请求可以进入扣减链路
```
但是进入扣减链路之后，它们仍然要排队等待同一行库存锁和少量账户锁。

于是结果变成：
```
失败率下降；
成功请求变多；
但请求等待锁的时间大幅增加；
整体 RT 和 P95 上升。
```
这就是为什么 V2 热点写的失败率更好，但 RT 更差。

这个结论很适合放进报告：
```
V2 并没有消除热点写锁，而是让更多请求有机会进入锁等待队列，因此错误率下降，但尾部延迟进一步放大。
```

## 13. V2 后续优化方向

基于 V2 的实验结果，下一阶段应进入 V3 写路径优化。

### 13.1 V3 优化目标

V3 应该重点解决：
```
Gift 库存热点行锁竞争
```
可选方案：
```
1. Redis 预扣库存；
2. Redis 库存令牌；
3. 库存分桶；
4. MQ 排队削峰；
5. 热点 Gift 限流；
6. 本地队列按 giftId 串行化。
```
***

### 13.2 推荐 V3 版本：Redis 预扣库存版

V3 可以先做：
```
V3 Redis Stock Pre-Deduct
```
链路调整为：
```
请求进入
↓
幂等初步判断
↓
Gift / LiveRoom 缓存读取
↓
Redis 预扣库存
↓
预扣成功后进入 MySQL 事务
↓
扣余额
↓
扣库存 / 或异步校准库存
↓
写订单
↓
写流水
```
V3 的实验目标：
```
1. 降低热点 giftId 对 MySQL 库存行的直接冲击；
2. 减少进入 MySQL 事务的无效请求；
3. 降低热点写场景下的 lock wait；
4. 降低 Hikari 连接占用时间；
5. 改善热点写场景的 P95 和失败率。
```
***

## 14. V2 报告最终结论

可以把下面这段作为 V2 报告的最终结论：
```
V2 Cache Preload 缓存前置版在分散写场景下取得了稳定收益。在 1000 个用户、100 个礼物、100 VU 的压测模型下，系统失败率保持 0%，RPS 从 V1 的 51.65/s 提升到 59.38/s，平均 RT 从 1.90s 降到 1.66s，P95 从 3.71s 降到 3.23s。这说明将 Gift 和 LiveRoom 这类读多写少数据前置到缓存，可以有效降低读路径数据库访问，并改善正常分散写场景下的吞吐和延迟。
但在热点写场景下，V2 并没有解决根因。虽然失败率从 V1 的 76.91% 降到 12.91%，但平均 RT 上升到 19.86s，P95 上升到 28.46s。结合链路分析，原因是缓存前置减少了读 DB 和连接池压力，使更多请求能够进入核心扣减链路，但这些请求最终仍然需要竞争同一行 Gift 库存和少量用户账户余额，因此请求被大量阻塞在 MySQL 行锁等待上。
因此，V2 的结论是：缓存前置可以优化读路径，在分散写场景下有效；但它不能解决高并发写热点问题。热点 Gift 库存扣减的核心瓶颈仍然在 MySQL 单行写锁竞争。下一阶段应进入 V3 写路径优化，引入 Redis 预扣库存、库存令牌、库存分桶或 MQ 削峰等方案，重点降低热点库存行对 MySQL 的直接冲击。
```
